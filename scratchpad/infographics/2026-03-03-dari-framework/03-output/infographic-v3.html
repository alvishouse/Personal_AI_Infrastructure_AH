The DARI Framework
DDiscover
AArchitect
RRun
IImprove
Discover
Map before you automate
Boundary Mapping Every task crossing a human-agent line
I/O Spec Define "done right" before you start
Define Success Can't detect failure without a definition
Architect
Design the handoffs
Human → Agent What does the agent need?
Agent → Human What must a human verify?
Agent → Agent Checkpoints between models in a chain
Agent → System Guardrails before data touches live systems
Every handoff: Clear · Verifiable · Recoverable
Run
Execute with visibility
Log Outputs Every run produces a traceable record
Flag Deviations Surface exceptions before they compound
Designated Owner Every agent needs an accountable human
Improve
Measure what's drifting
Capability Drift The model itself changes
Performance Drift Outputs decline without any model change
Context Drift Business changed, but the agent didn't
Review cadence: Daily · Weekly · Monthly
Before You Delegate to an Agent, Ask:
→Have I mapped every task crossing a human-agent boundary?
→Can I define what "done right" looks like for every output?
→Does every handoff have a verifiable, recoverable checkpoint?
→Is there a traceable log for every agent run?
→Is there an accountable human for this agent — the same way a machine has an operator?
The Bottom Line
Discover
Map the work before you automate it
Architect
Design the handoffs, not just the tasks
Run + Improve
Execute with visibility. Measure what drifts.
DARI isn't a replacement for Lean — it's Lean extended to the delegation problem.
